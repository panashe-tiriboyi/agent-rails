# Agent Rails

This kit helps developers start new projects, add AI resources to existing repositories, or prepare migration work with guided rails instead of ad hoc prompts.

It pairs with the `agent-rails` skill. The README is the human entrypoint; the files in `prompts/` are copyable prompt modules; the `.ai/` folder is a portable scaffold that can be copied into a target repo or generated by the skill.

## What This Kit Creates

- A project primer for agents.
- Thin adapter files for common AI coding tools.
- A context map for code, docs, tests, deployment, decisions, and known issues.
- A requirements flow that requires user approval before implementation.
- Prompt modules for analysis, planning, tests, review, and migrations.
- Guardrails for clean architecture, separation of concerns, dependency hygiene, security checks, and evidence-based claims.
- A bundled `agent-rails` skill zip for portable installation.

## Quick Start

1. Pick your goal: new project, existing project, or migration.
2. Copy the matching prompt from the Prompts section below into your AI coding agent.
3. Answer the agent's questions.
4. Review the proposed AI-resource plan.
5. Approve file generation only after the plan matches your intent.
6. Scaffold the repo: either let the agent generate the files from the approved plan, or run the bundled `agent-rails` skill for deterministic generation (see below).

## Kit vs. Bundled Skill Pack

The bundled skill generates a minimal portable core by design: `AGENTS.md`, `CODEX.md`, the base `.ai/` scaffold with core context primers and prompts, and requirements intake. This kit is the full reference scaffold — it additionally carries the `CLAUDE.md`, `GEMINI.md`, `CURSOR.md`, and Copilot adapters plus extended context primers (architecture, testing, security, operations, migration, requirements flow). Copy those from the kit into a target repo when the project needs more than the core.

## Install The Skill

This kit includes `skills/agent-rails.zip`.

To install it manually:

1. Unzip `skills/agent-rails.zip`.
2. Copy the unzipped `agent-rails` folder into your agent or Codex skills directory.
3. Restart or reload the agent environment if required by the platform.
4. Run the generator in dry-run mode before applying changes.

Suggested command after plan approval:

```powershell
python <path-to-agent-rails>\scripts\generate_ai_resources.py --target <repo-path> --mode existing --tier standard --dry-run
python <path-to-agent-rails>\scripts\generate_ai_resources.py --target <repo-path> --mode existing --tier standard --apply
```

```bash
python <path-to-agent-rails>/scripts/generate_ai_resources.py --target <repo-path> --mode existing --tier standard --dry-run
python <path-to-agent-rails>/scripts/generate_ai_resources.py --target <repo-path> --mode existing --tier standard --apply
```

For migration setup, add:

```powershell
--resource-mode migration
```

Note on flags: `--mode new|existing` describes the state of the target directory and controls merge behavior; `--resource-mode new-project|existing-project|migration` describes the shape of the resource pack and matches the three workflow modes in this kit's prompts. They combine — for example, `--mode existing --resource-mode migration` adds migration resources to an existing repo without overwriting it.

## Prompts

Pick the one prompt that matches your goal and copy it into the agent you want to use. Each prompt tells the agent to generate only what that goal needs and to remove scaffold files that do not apply.

### Prompt 1: New Project

```text
You are helping me create project-specific AI resources for a brand-new project.

Goal:
Generate an AI-resource plan and, only after my approval, a resource pack that helps future AI agents understand the project, ask the right questions, flag risks, preserve architecture quality, and avoid implementing unapproved work.

Core agent archetypes:
- Always consider: orchestrator, architect, requirements, documentation, QA, reviewer.
- Add only when project fit requires them: auth, backend, database, frontend, data/AI, DevOps, security, domain specialists.
- Do not create specialist agents that do not match the project.

Default engineering rails:
- Prefer clean architecture for the chosen stack.
- Enforce separation of concerns, testability, dependency hygiene, and maintainable boundaries.
- Prefer latest supported stable versions and secure dependencies.
- Flag security, privacy, authorization, data, infrastructure, and dependency risks.
- Allow prototype shortcuts only if I explicitly approve them; still record the risks.
- Requirements that lead to implementation must be approved by me before implementation begins.

First response instructions:
Do not generate files yet. Start with an interactive questionnaire. Ask concise questions that materially affect the output. Cover:
- Project purpose, audience, and success criteria.
- Stack, platform, language, framework, and deployment target if known.
- Whether this is prototype, production, learning, internal tool, regulated, customer-data, or security-sensitive.
- Desired governance level: lightweight, standard, or strict.
- Approval authority for requirements, target behavior, and architecture decisions.
- Verification commands or expected test strategy.
- Which specialist agents are needed.
- Any known constraints, risks, or non-goals.

After the questionnaire:
Produce an AI-resource plan with the recommended agent set, context primers, prompt modules, requirement and approval flow, test and review expectations, risks to flag, files to generate, and open decisions.

Only after I approve the plan:
Generate the AI resource pack. Because this is a new project, do not create migration resources: skip migration prompts, migration context primers, and docs/migration. Remove any scaffold files or folders that do not apply to this project. Use placeholders only for unresolved decisions and record them as open decisions.
```

### Prompt 2: Existing Project

```text
You are helping me add project-specific AI resources to an existing codebase.

Goal:
Inspect the current workspace, then generate an AI-resource plan and, only after my approval, a resource pack that fits this project without overwriting useful existing guidance.

Core agent archetypes:
- Always consider: orchestrator, architect, requirements, documentation, QA, reviewer.
- Add only when project fit requires them: auth, backend, database, frontend, data/AI, DevOps, security, domain specialists.
- Do not create specialist agents that do not match the project.

Default engineering rails:
- Prefer clean architecture for the existing stack.
- Enforce separation of concerns, testability, dependency hygiene, and maintainable boundaries.
- Flag security, privacy, authorization, data, infrastructure, and dependency risks.
- Requirements that lead to implementation must be approved by me before implementation begins.
- Documentation claims are not verified behavior until checked against code, tests, runtime output, or approved decisions.

First response instructions:
Do not generate files yet. First inspect the workspace: existing root guidance (AGENTS.md, CLAUDE.md, README), package manifests, docs, tests, and deployment files. Then ask concise questions that inspection could not answer. Cover:
- Project purpose, audience, and current state.
- Whether this is prototype, production, internal tool, regulated, customer-data, or security-sensitive.
- Desired governance level: lightweight, standard, or strict.
- Approval authority for requirements, target behavior, and architecture decisions.
- Source-of-truth docs and existing requirements.
- Verification commands or expected test strategy.
- Which specialist agents are needed.
- Any known constraints, risks, or non-goals.

After inspection and the questionnaire:
Produce an AI-resource plan with the recommended agent set, context primers, prompt modules, requirement and approval flow, test and review expectations, risks to flag, files to generate, files to preserve, and open decisions.

Only after I approve the plan:
Generate the AI resource pack. Preserve existing files by default; do not overwrite existing AGENTS.md, CODEX.md, .ai files, or requirements docs without my explicit approval. Do not create migration resources: skip migration prompts, migration context primers, and docs/migration. Remove any scaffold files or folders that do not apply to this project. Use placeholders only for unresolved decisions and record them as open decisions.
```

### Prompt 3: Migration

```text
You are helping me create project-specific AI resources for a migration project.

Goal:
Map the old system, the migration target, docs, behavior, data, tests, and risks into evidence-labeled AI resources before any implementation. Generate an AI-resource plan and, only after my approval, the resource pack.

Core agent archetypes:
- Always consider: orchestrator, architect, requirements, documentation, QA, reviewer, migration analyst.
- Add only when project fit requires them: auth, backend, database, frontend, data/AI, DevOps, security, domain specialists.

Default engineering rails:
- Prefer clean architecture for the target stack.
- Enforce separation of concerns, testability, dependency hygiene, and maintainable boundaries.
- Flag security, privacy, authorization, data, infrastructure, and dependency risks.
- Requirements that lead to implementation must be approved by me before implementation begins.
- Documentation claims are not verified behavior until checked against code, tests, runtime output, or approved decisions.
- Keep migration evidence separate from target decisions.

First response instructions:
Do not generate files yet. Start with an interactive questionnaire. Ask concise questions that materially affect the output. Cover:
- What is being migrated: system, stack, data, workflows, integrations.
- Where the old system's source of truth lives: code, docs, runtime, people.
- The migration target: stack, platform, deployment, and success criteria.
- Whether behavior parity is required, and who approves accepted differences.
- Desired governance level: lightweight, standard, or strict.
- Approval authority for requirements, target behavior, and architecture decisions.
- Verification commands or expected test strategy for both old and new systems.
- Known risks, constraints, and non-goals.

After the questionnaire:
Produce an AI-resource plan with the recommended agent set, migration context primers, prompt modules, an evidence-labeling approach that separates assumptions, documentation claims, observed behavior, tests, and approved decisions, the requirement and approval flow, risks to flag, files to generate, and open decisions.

Only after I approve the plan:
Generate the full AI resource pack including migration prompts, migration context primers, and docs/migration. Preserve existing files by default. Remove any scaffold files or folders that do not apply to this project. Use placeholders only for unresolved decisions and record them as open decisions.
```

## Scaffold

The `.ai/` folder here is intentionally scaffold-first. It exists so projects can start with the right shape:

- `.ai/agents`
- `.ai/context`
- `.ai/prompts`
- `.ai/skills`

Use `.gitkeep` markers to preserve empty folders in Git.

Root adapter files:

- `AGENTS.md`
- `CODEX.md`
- `CLAUDE.md`
- `GEMINI.md`
- `CURSOR.md`
- `.github/copilot-instructions.md`

Practical docs scaffold:

- `docs/requirements`
- `docs/architecture`
- `docs/testing`
- `docs/security`
- `docs/operations`
- `docs/migration`

## Safety Defaults

- Ask before implementation.
- Preserve existing guidance.
- Keep project facts in one canonical place.
- Treat security shortcuts as explicit decisions, not accidents.
- Separate assumptions, documentation claims, observed behavior, tests, and approved decisions.
